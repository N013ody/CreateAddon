# 2026-06-04 航空学与 Create 附属经验整理

## 目标

按项目协作要求继续学习 Create Aeronautics / Simulated Project，以及其他成熟 Create 附属的制作细节，把经验转化为计算学后续设计和实现时可复用的规则。

## 研究对象

- Create Aeronautics / Simulated Project：重点看物理子世界、推进器、陀螺推进器、Ponder、tooltip 和 SubLevel 装配/拆解。
- Create Big Cannons：重点看复杂机构如何拆成控制器、contraption entity、接口、兼容注册表、渲染和 Ponder。
- Create Crafts & Additions：重点看外部系统如何作为适配层桥接到 Create 动力网络。

## 设计整理

这次学习强化了一个方向：计算学应把“计算”和“控制”做成通用机械能力，而不是直接做一个黑盒机器。

航空学的经验说明，飞行和物理不是单方块行为，而是物理子世界、力、质量、姿态、推进器、传感、异常和可视化共同构成的系统。计算学未来要接航空学，应先提供通用控制信号、传感输入、机械 PID、稳定性和延迟模型，再通过可选适配器输出到航空学。

Big Cannons 的经验说明，大型 Create 机构必须拆层：控制器管交互和动力，contraption entity 管移动结构，接口管控制协议，渲染单独处理视觉，Ponder 单独解释使用流程。计算学以后做分析机、差分机、飞控计算机，都应继续保持模块化。

Crafts & Additions 的经验说明，外部系统可以桥接，但不能吞掉 Create 核心体验。计算学不应把 FE/RF 作为核心能量，而应继续使用转速、应力、组件能力和世界内搭建作为主玩法。

## 后续建议

1. 做“控制信号总线”前，先定义通用控制数据：pitch、yaw、roll、throttle、brake、target vector、error、confidence。
2. 做航空学兼容前，先让计算学核心在没有 Aeronautics 时也能完整运行。
3. 做任何新玩家可见方块时，必须同时准备 Shift tooltip、W/Ponder、各面作用说明、语言键和制作记录。
4. 若以后需要 mixin，先写小型实验和隔离兼容层，不能直接侵入核心计算逻辑。
